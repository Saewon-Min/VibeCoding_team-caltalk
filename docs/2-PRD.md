## 1. 문서 개요

### 1.1 목적
본 문서는 "Team CalTalk"의 제품 요구사항을 정의한다. 도메인, 용어, 역할, 비즈니스 규칙은 이미 `1-domain-definition.md`(v1.2)에서 확정되었으며, 본 PRD는 그 위에 "왜 만드는가(비즈니스 목표), 누구를 위한 것인가(목표 사용자), 무엇을 언제까지 만드는가(범위와 일정), 무엇으로 만드는가(기술 스택)"를 결합하여 실행 가능한 제품 계획으로 구체화한다.

### 1.2 참고 문서
- `docs/1-domain-definition.md` (v1.2) — 용어(TERM-xx), 액터(ROLE-xx), 엔티티(ENT-xx), 비즈니스 규칙(BR-xx), 요구사항 추적 매트릭스(REQ-xx)
- `docs/3-user-scenarios.md` (v1.0) — 시나리오 SC-01~SC-12

### 1.3 문서 관계
본 문서의 기능 요구사항은 도메인 정의서의 ID 체계를 그대로 인용한다. 도메인 정의서가 "무엇이 참인가(개념/규칙)"를 정의한다면, 본 PRD는 "무엇을 언제까지 만들 것인가(우선순위/일정/비기능 요건)"를 정의한다. 상세 DB 스키마, API 명세, 화면 설계는 본 문서의 범위 밖이며 후속 산출물에서 다룬다.

## 2. 제품 개요

Team CalTalk은 4~10인 규모의 소규모 팀을 위한 팀 캘린더 + 팀 채팅 통합 도구다. 팀장은 팀 일정을 생성·수정·삭제하는 배타적 권한을 가지고, 팀원은 캘린더를 조회하며 필요 시 채팅을 통해 일정 변경을 "요청"한다. 이 요청과 그에 대한 팀장의 승인/거절 처리 결과가 모두 해당 일자의 채팅 이력에 자동으로 남는다.

경쟁 제품들은 캘린더와 대화 도구가 분리되어 있어 "왜 이 일정이 이렇게 바뀌었는지"에 대한 근거가 유실되기 쉽다. Team CalTalk의 핵심 가치 제안은 **"일정 변경의 근거를 채팅 이력으로 자동 보존한다"**는 점이다. 별도의 변경 이력 화면이나 감사 로그 기능을 추가로 만들지 않아도, 팀이 원래 사용하던 채팅이 곧 일정 변경의 승인 기록이자 근거 자료가 된다.

## 3. 목표 사용자 및 시장

### 3.1 목표 사용자
- 팀 규모: 4~10명 수준의 소규모 팀 (그 이상/이하 규모의 상세 페르소나는 이번 범위에서 정의하지 않음)
- 대상 업종/조직 유형: IT 개발팀, 스타트업, 소프트웨어 하우스 등 일정 조율과 협업 대화가 빈번하고 팀장 중심의 일정 통제가 필요한 조직
- 역할: 도메인 정의서 ROLE-01(팀장), ROLE-02(팀원)를 그대로 따름

### 3.2 비즈니스 목표
- 단기(제품 검증): 위 목표 사용자군이 실제 업무에서 캘린더와 채팅을 하나의 화면에서 함께 쓸 수 있는 최소 기능 제품(MVP)을 확보한다.
- 장기(확장 목표): 3,000개 이상의 팀이 동시에 프로젝트를 진행할 수 있는 서비스로 성장시킨다. 이는 제품/비즈니스 차원의 지향점이며, 본 PRD가 다루는 1차 개발 범위(5일/1인)에서 실제로 검증되는 목표가 아니라는 점을 5장과 8장, 11장에서 명확히 구분한다.

## 4. 경쟁 환경 및 차별화 포인트

| 구분 | 일반 캘린더/일정관리 도구 | 일반 팀 채팅 도구 | Team CalTalk |
|---|---|---|---|
| 일정 관리 | 강함 | 없거나 약함 | 팀장 중심 권한 모델로 제공 |
| 팀 대화 | 없거나 약함 | 강함 | 일정과 같은 화면/맥락에서 제공 |
| 일정 변경 근거 추적 | 별도 변경 이력/감사로그 기능 필요 | 대화는 남지만 일정과 연결되지 않음 | 변경 요청·승인·거절이 자동으로 해당 일자 채팅 이력에 귀속(BR-13) |
| 권한 통제 | 대체로 전원 동일 권한 | 해당 없음 | 팀 일정 쓰기 권한을 팀장에게 배타적으로 귀속(BR-02, BR-03), 팀원은 채팅 기반 요청만 가능(BR-04) |

차별화 포인트는 신규 기능을 추가하는 방식이 아니라, 팀이 이미 쓰는 채팅이라는 매체를 일정 변경의 승인 근거로 그대로 활용한다는 점에 있다. 이는 도메인 정의서의 BR-06(채팅 이력 영구 보존), BR-13(처리 결과의 채팅 이력 기록)에 의해 뒷받침된다.

## 5. 목표 및 성공지표(KPI)

MVP(5일/1인 개발 범위)에서 측정 가능한 지표와, 3,000팀 규모 확장이라는 장기 비즈니스 목표에 대한 지표는 성격이 다르므로 명확히 분리한다.

### 5.1 MVP 단계 성공지표 (5일 종료 시점 측정 가능)

| 지표 | 목표 | 측정 방법 |
|---|---|---|
| 핵심 유스케이스 완결성 | 6.2/6.3(도메인 정의서) 시나리오가 오류 없이 end-to-end 동작 | 팀장/팀원 계정으로 수동 시나리오 워크스루 |

| API 응답시간(p95) | 별도 정의 예정 | 스케일 검증 단계에서 목표치 설정 |

**중요**: 5일/1인 개발이라는 제약 하에서 3,000팀 동시 운영에 대한 성능 테스트, 튜닝, 인프라 확장 설계까지 완료하는 것은 이번 범위에 포함되지 않는다. MVP는 "기능이 규칙대로 올바르게 동작하는가"를 검증하는 데 집중하고, 확장성은 아키텍처 설계 시 무리한 구조적 걸림돌을 만들지 않는 수준으로만 고려한다(9장 참조).

## 8. 비기능 요구사항

### 8.1 성능 및 확장성
- MVP 목표: 4~10인 규모의 단일 팀 사용 시 캘린더 조회, 채팅 전송, 일정 변경 요청 처리가 체감상 지연 없이 동작해야 한다(수동 QA 기준).
- 확장성 설계 원칙: 3,000팀 동시 운영이라는 장기 비즈니스 목표(5장)를 고려하여, 팀 단위로 데이터가 명확히 분리되는 스키마 설계(ENT-02 Team을 모든 주요 엔티티의 소유 기준으로 유지, BR-07)와 같이 향후 수평 확장을 어렵게 만드는 구조적 결정은 피한다.
- MVP 범위에서 하지 않는 것: 실제 3,000팀 규모의 부하 테스트, 캐싱 계층 도입, DB 샤딩/복제, 오토스케일링 등 인프라 확장 작업은 수행하지 않는다. 이는 별도 후속 과제로 남긴다.

### 8.2 보안(인증/인가 최소 요건)
- 모든 API는 인증된 사용자만 호출할 수 있어야 한다(BR-01).
- 모든 팀/일정/채팅 리소스 조회·조작 API는 요청자가 해당 팀에 소속된 사용자인지 서버 측에서 검증해야 하며, 소속되지 않은 팀의 리소스는 응답에 포함하지 않는다(BR-16).
- 팀 일정 생성/수정/삭제 API는 요청자가 해당 팀의 팀장(ROLE-01)인지 서버 측에서 검증해야 한다(BR-02, BR-03).
- 일정 변경 요청 제기는 요청자가 대상 일정의 참여자(ENT-05)인지 서버 측에서 검증해야 한다(BR-10).
- 일정 변경 요청 승인/거절, 취소는 각각 팀장 본인/요청자 본인 여부를 서버 측에서 검증해야 한다(BR-05, BR-12).
- 비밀번호 등 인증정보는 평문으로 저장하지 않는다. 세션/토큰 기반 인증 방식은 구현 단계에서 결정한다.

### 8.3 가용성
- MVP 단계에서 별도의 SLA 수치 목표는 설정하지 않는다. 1인 개발/5일 일정 특성상 이중화, 무중단 배포, 재해복구 등은 이번 범위에 포함하지 않는다.

### 8.4 반응형 UI
- 웹 브라우저를 1차 플랫폼으로 하며, 데스크톱 해상도를 기준으로 우선 개발하되 모바일 브라우저 폭에서도 캘린더/채팅 핵심 화면이 깨지지 않고 사용 가능해야 한다(반응형 레이아웃, 별도 네이티브 앱 없음).

### 8.5 접근성
- 본 MVP 범위에서 접근성(스크린리더 지원, 키보드 내비게이션, WCAG 준수 등)은 명시적으로 고려하지 않는다.

## 9. 기술 아키텍처 개요

본 절은 상위 수준 구성 방향만 제시하며, 상세 DB 스키마와 API 명세는 본 문서의 범위 밖으로 후속 산출물에서 다룬다.

| 계층 | 기술 | 역할 |
|---|---|---|
| 프론트엔드 | React | 캘린더(월/주/일) 및 채팅 UI 렌더링, 반응형 레이아웃 처리, 백엔드 API 호출 |
| 백엔드 | Node.js + Express | 인증/인가, 도메인 규칙(BR-01~BR-16) 검증, 비즈니스 로직 처리, REST API 제공 |
| 데이터베이스 | PostgreSQL | 도메인 정의서 5.2절 엔티티(ENT-01~ENT-07)에 대응하는 데이터 영속화, 관계형 제약을 통한 데이터 무결성 확보 |

- 프론트엔드-백엔드 간 통신은 REST API 기반을 전제로 한다. 채팅의 실시간성 구현 방식(폴링 주기, 또는 향후 WebSocket 등 도입 여부)은 본 PRD에서 확정하지 않으며 구현 단계의 결정 사항으로 남긴다.
- 3,000팀 규모로 확장할 경우 캐싱, 메시지 브로커, 서비스 분리 등의 도입을 검토할 수 있으나, 이는 향후 확장 검토 대상일 뿐 이번 MVP의 아키텍처에 포함되지 않는다.
- 인프라/배포 환경은 1인 개발 예산에 맞춰 별도의 유료 관리형 서비스에 크게 의존하지 않는 구성을 기본 전제로 한다.

## 10. 일정 및 마일스톤 (5일 x 1인 개발)

5일/1인이라는 제약 안에서 도메인 정의서의 핵심 규칙(BR-01~BR-16)을 모두 다루는 것을 목표로 하되, 일정 변경 요청과 관련된 규칙(BR-10~BR-13)은 논리적으로 서로 얽혀 있어(요청 제기 → 승인/거절 → 자동 거절 처리 → 취소 → 결과 기록) 하루 안에 함께 구현하는 것이 현실적이며 무리 없이 5일 내에 수용 가능하다고 판단한다. 다만 이력 검색, 알림 UI 고도화 등 편의 기능은 Should/Could로 미룬다(7장 참조).

| Day | 주요 작업 | 대응 Must-have | 산출물 |
|---|---|---|---|
| Day 1 | 프로젝트 구조 세팅, DB 스키마 설계 및 마이그레이션(ENT-01~07 대응), 인증(회원가입/로그인), 팀 생성(자동 팀장 등록), 팀원 추가(이메일 검색 기반 즉시 추가), 팀-사용자 소속/역할 부여, 팀 경계 접근 제어(소속 팀 리소스만 조회) 기초 구현 | BR-01, BR-08, BR-09(기초 검증), BR-14, BR-15, BR-16 | 인증 API, 팀/팀원 관리 API, 초기 DB 스키마 |
| Day 2 | 팀 일정 CRUD(팀장), 일정 참여자 지정, 캘린더 월/주/일 조회 UI(팀장/팀원 공통), 권한 검증(팀원 쓰기 차단) | BR-02, BR-03 | 캘린더 화면, 일정 CRUD API |
| Day 3 | 팀 채팅 메시지 작성/조회, 일자 단위 이력 그룹핑 및 영구 보존, 캘린더-채팅 연동 레이아웃(기초) | BR-06 | 채팅 화면, 메시지 API |
| Day 4 | 일정 변경 요청 제기(참여자 검증), 팀장 승인/거절 처리, 동일 일정 복수 대기 요청 자동 거절 처리, 요청자 본인 취소, 처리 결과의 채팅 이력 기록 | BR-04, BR-05, BR-10, BR-11, BR-12, BR-13 | 일정 변경 요청 API 및 화면 |
| Day 5 | 통합 테스트(BR-01~BR-16 체크리스트 기반 수동 검증), 반응형 레이아웃 점검, 버그 픽스, 배포 | 5.1절 MVP 성공지표 전체 | 배포된 MVP, 검증 체크리스트 결과 |

리스크에 대비해, Day 5에 예정된 버그 픽스 시간이 부족할 경우 우선적으로 축소할 항목은 Should-have(통합 레이아웃 완성도, 상태 시각화, 인앱 알림)이며, Must-have(BR-01~BR-16)는 최후까지 유지하는 것을 원칙으로 한다.

## 12. Out of Scope / 향후 로드맵

아래 항목은 이번 MVP 범위에 포함되지 않으며, 제품이 성장할 경우 향후 검토 대상이다.

| 항목 | 비고 |
|---|---|
| Google Calendar, Slack 등 외부 서비스 연동 | 팀이 이미 쓰는 외부 도구와의 일정/대화 동기화. 인증 연동(OAuth) 및 API 사용량 관리 등 추가 검토 필요 |
| 네이티브 모바일 앱(iOS/Android) | 반응형 웹으로 우선 대응 후, 사용자 요구가 확인되면 별도 검토 |
| 접근성(a11y) 표준 준수 | WCAG 등 기준 적용 여부 및 범위는 별도 과제 |
| 알림 채널의 세부 구현(이메일, 푸시, SMS 등) | 도메인 정의서 8장과 동일. "알림 필요성"만 인지하며 발송 수단은 범위 외 |
| 조직/부서 등 팀 상위 계층 구조 | 현재는 팀을 최상위 독립 단위로 취급 |
| 시스템 관리자(운영자) 역할 및 관리자 콘솔 | 현재는 팀장/팀원 2개 역할만 정의 |
| 다국어 지원, 타임존 세부 처리 | 구현 단계 및 해외 진출 검토 시점에 재논의 |
| 결제/구독 관리 | 유료화 모델 수립 시 별도 문서화 |
| 3,000팀 규모 성능/확장성 검증 | 부하 테스트, 캐싱/복제/샤딩 등 인프라 확장, SLA 수치화는 스케일 검증 단계의 별도 과제 |
| 채팅 부가 기능(파일 첨부, 이모지 반응, 메시지 수정/삭제) | 텍스트 기반 대화와 일정 변경 요청 흐름만 우선 제공 |
