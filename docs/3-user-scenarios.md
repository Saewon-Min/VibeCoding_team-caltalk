## 1. 문서 개요

### 1.1 목적
본 문서는 `1-domain-definition.md`(v1.1)의 도메인 모델·비즈니스 규칙과 `2-PRD.md`(v1.0)의 기능 우선순위를 기반으로, 실제 개발과 QA가 화면·API 설계 시 바로 참고할 수 있는 수준의 구체적인 사용자 시나리오를 정의한다. 도메인 정의서 6장의 스켈레톤 유스케이스(팀장 1~5단계, 팀원 1~5단계)를 예시 데이터를 사용한 서술형 스텝 시나리오로 확장한 것이다.

### 1.2 참고 문서
- `docs/1-domain-definition.md` (v1.2) — 용어(TERM-xx), 액터(ROLE-xx), 엔티티(ENT-xx), 비즈니스 규칙(BR-xx)
- `docs/2-PRD.md` (v1.1) — 7장 기능 요구사항(MoSCoW), Must-have 14개 항목

### 1.3 시나리오 표기법
각 시나리오는 다음 항목으로 구성된다.
- **시나리오 ID / 제목**
- **관련 액터**: 도메인 정의서 4.1절 ROLE-xx
- **관련 PRD Must-have**: `2-PRD.md` 7장 표의 해당 행(우선순위 Must)
- **관련 BR**: 도메인 정의서 7장 BR-xx
- **사전조건(Precondition)**: 시나리오 시작 시점에 참이어야 하는 상태
- **기본 흐름(Main Flow)**: 번호가 매겨진 단계. "사용자 행동"과 "시스템 반응"을 구분해 서술
- **대체/예외 흐름(Alternative/Exception Flow)**: 기본 흐름에서 분기되는 실패·거부·대안 케이스. 특히 BR-xx 위반 시도가 시스템에 의해 차단되는 경로를 최소 1개 포함
- **사후조건(Postcondition)**: 시나리오 종료 시점에 성립하는 상태

## 3. 시나리오 상세

### SC-01. 회원가입 및 로그인, 미인증 접근 차단

- **관련 액터**: ROLE-01(팀장), ROLE-02(팀원)
- **관련 PRD Must-have**: 회원가입/로그인(인증)
- **관련 BR**: BR-01

**사전조건**
- 김철수는 아직 Team CalTalk에 계정이 없다.

**기본 흐름**
1. 김철수가 회원가입 화면에서 이름 "김철수", 이메일 "kim.leader@techteam.io", 비밀번호를 입력하고 가입을 요청한다.
2. 시스템이 이메일 형식과 중복 여부를 검증하고, 비밀번호를 평문으로 저장하지 않는 방식으로 처리한 뒤 User(ENT-01, TERM-01) 레코드를 생성하고 가입 완료를 응답한다.
3. 김철수가 로그인 화면에서 동일한 이메일/비밀번호를 입력하고 로그인을 요청한다.
4. 시스템이 자격 증명을 검증하고(TERM-10 인증), 인증 세션/토큰을 발급한 뒤 팀 목록 화면으로 이동시킨다. 이후 모든 API 호출에는 이 인증 토큰이 필요하다.

**대체/예외 흐름**
- **E1 (자격 증명 오류)**: 3단계에서 비밀번호를 잘못 입력하면 시스템은 401 응답과 함께 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지를 반환하고 로그인 화면에 머무른다.
- **E2 (미인증 접근 차단, BR-01)**: 이서연이 아직 로그인하지 않은 상태에서 브라우저 주소창에 테크팀 캘린더 URL을 직접 입력해 접근을 시도한다. 시스템은 요청에 유효한 인증 토큰이 없음을 확인하고, 팀/일정/채팅 데이터를 전혀 포함하지 않은 401 Unauthorized 응답을 반환하며 클라이언트를 로그인 화면으로 리다이렉트한다.

**사후조건**
- 김철수는 인증된 세션을 보유하며 이후 시나리오(SC-02 이하)를 진행할 수 있다.
- 미인증 사용자는 어떠한 팀/일정/채팅 리소스에도 접근하지 못한다.

### SC-02. 팀 생성 및 팀원 추가

- **관련 액터**: ROLE-01(팀장), ROLE-02(팀원)
- **관련 PRD Must-have**: 팀 생성(생성자 자동 팀장 등록) / 팀원 초대(이메일 검색 기반 즉시 추가, 발송/수락 절차 없음) / 팀-사용자 소속 및 역할 부여, 한 사용자가 여러 팀에 다른 역할로 소속 가능
- **관련 BR**: BR-08, BR-09, BR-14, BR-15

**사전조건**
- 김철수는 로그인한 상태이며(SC-01 완료), 아직 어떤 팀에도 소속되어 있지 않다.
- 이서연, 박준영, 최유진은 이미 Team CalTalk에 회원가입을 완료한 상태다(BR-14의 전제조건).

**기본 흐름**
1. 김철수가 "새 팀 만들기"를 선택하고 팀 이름 "테크팀"을 입력해 생성을 요청한다.
2. 시스템이 Team(ENT-02) 레코드를 생성하고, 생성자 김철수를 해당 팀의 TeamMembership(ENT-03)에 역할=팀장으로 자동 등록한다(BR-15). 이 시점부터 테크팀은 BR-09가 요구하는 "최소 1인의 팀장"을 보유한다.
3. 김팀장이 팀원 관리 화면에서 이메일 "lee.member@techteam.io"로 사용자를 검색한다. 이미 가입된 이서연이 검색 결과에 노출된다.
4. 김팀장이 검색된 이서연을 "팀원으로 추가"하면, 시스템은 즉시 TeamMembership(ENT-03)을 역할=팀원으로 생성한다(BR-14). 별도의 초대 발송이나 수락 대기 절차는 없다. 같은 방식으로 박준영(park.member@techteam.io), 최유진(choi.member@techteam.io)도 검색해 추가한다.
5. 이서연, 박준영, 최유진이 각자 로그인하면 팀 목록에 "테크팀"이 즉시 표시되고, 각자의 역할은 팀원으로 확인된다.

**대체/예외 흐름**
- **A1 (BR-08, 사용자별 다중 팀·다른 역할)**: 김철수는 별도로 "디자인팀"(팀장 정다은)에는 팀원으로 소속되어 있다. 로그인 후 팀 전환 화면에서 테크팀을 선택하면 팀장 권한 UI가, 디자인팀을 선택하면 팀원 권한 UI가 노출된다. 즉 역할은 사용자 고유 속성이 아니라 팀-사용자 소속 관계에 종속된다.
- **E1 (팀원의 추가 권한 없음)**: 이서연이 팀원 관리 화면(또는 팀원 추가 API)에 접근해 다른 사용자를 추가하려고 시도한다. 시스템은 이서연이 테크팀의 팀장이 아님을 확인하고 403 Forbidden "팀원 추가 권한이 없습니다"를 반환하며 TeamMembership을 생성하지 않는다.
- **E2 (미가입 사용자 검색, BR-14)**: 김팀장이 아직 가입하지 않은 이메일 "new.hire@techteam.io"로 검색한다. 시스템은 해당 이메일의 User가 존재하지 않음을 확인하고 "가입된 사용자를 찾을 수 없습니다. 먼저 회원가입이 필요합니다"를 반환하며 추가를 진행하지 않는다.

**사후조건**
- 테크팀에는 팀장 1명(김철수)과 팀원 3명(이서연, 박준영, 최유진)이 소속되어 있다.

### SC-03. 팀장의 팀 일정 생성

- **관련 액터**: ROLE-01(팀장), ROLE-02(팀원)
- **관련 PRD Must-have**: 팀 일정 생성/수정/삭제(팀장 전용) / 일정 참여자 지정
- **관련 BR**: BR-02, BR-03, BR-07

**사전조건**
- 테크팀이 존재하고 김철수(팀장), 이서연·박준영·최유진(팀원)이 소속되어 있다(SC-02 완료).

**기본 흐름**
1. 김팀장이 캘린더 화면에서 2026-07-14(화) 날짜를 선택하고 "일정 추가"를 클릭한다.
2. 제목 "스프린트 계획 회의", 설명 "3주차 스프린트 목표 및 작업 분배", 시작 2026-07-14 10:00, 종료 2026-07-14 11:00을 입력한다.
3. 참여자 지정 항목에서 이서연, 박준영을 선택한다.
4. "저장"을 클릭하면 시스템은 요청자가 테크팀의 팀장인지 서버 측에서 검증하고(BR-02), 검증을 통과하면 Schedule(ENT-04)을 생성하고 ScheduleParticipant(ENT-05) 레코드를 이서연·박준영에 대해 각각 생성한다.
5. 캘린더에 새 일정이 표시되며, 이서연·박준영의 캘린더에도 동일한 일정이 조회된다. 이 일정은 테크팀에만 귀속되며 다른 팀과 공유되지 않는다(BR-07).

**대체/예외 흐름**
- **E1 (BR-02/BR-03 위반, 팀원의 일정 생성 시도 거부)**: 이서연이 일정 생성 API를 직접 호출해 새 일정을 만들려고 시도한다(UI 상에는 생성 버튼이 노출되지 않는다는 전제). 시스템은 이서연이 테크팀 팀장이 아님을 확인하고 403 Forbidden "팀 일정 생성 권한이 없습니다"를 반환하며 Schedule을 생성하지 않는다.

**사후조건**
- 테크팀 캘린더에 "스프린트 계획 회의"(07-14, 10:00~11:00) 일정이 존재하고 참여자는 이서연, 박준영이다.

### SC-04. 캘린더 월/주/일 단위 조회

- **관련 액터**: ROLE-01(팀장), ROLE-02(팀원)
- **관련 PRD Must-have**: 팀 일정 월/주/일 단위 조회(팀장/팀원 공통, 소속 팀 데이터만 접근 가능)
- **관련 BR**: BR-03, BR-16

**사전조건**
- "스프린트 계획 회의"(07-14) 일정이 존재한다(SC-03 완료).

**기본 흐름**
1. 이서연이 캘린더 화면에서 "월간" 보기를 선택하면, 2026년 7월 달력 위에 "스프린트 계획 회의"를 포함한 테크팀의 일정들이 날짜별로 표시된다.
2. 이서연이 "주간" 보기로 전환하면, 07/13(월)~07/19(일) 한 주의 일정이 시간대별 타임라인으로 표시된다.
3. 이서연이 "일간" 보기로 전환해 07/14를 선택하면, 해당 날짜의 일정이 시간 단위로 표시되며 "스프린트 계획 회의(10:00~11:00)"를 확인할 수 있다.
4. 이서연이 일정 항목을 클릭하면 제목·시간·참여자 등 상세 정보가 읽기 전용으로 표시될 뿐, 수정 화면으로는 진입할 수 없다(BR-03).
5. 김팀장도 동일한 월/주/일 화면에서 조회할 수 있으며, 팀장에게는 추가로 수정/삭제 버튼이 노출된다.

**대체/예외 흐름**
- **E1 (BR-16, 팀 경계를 벗어난 조회 차단)**: 테크팀에 소속되지 않은 정다은(디자인팀 팀장)이 테크팀 캘린더 URL에 직접 접근을 시도한다. 시스템은 정다은이 테크팀의 TeamMembership을 가지고 있지 않음을 확인하고 403 Forbidden "해당 팀 소속이 아닙니다"를 반환하며 일정 데이터를 노출하지 않는다.

**사후조건**

### SC-07. 팀장의 일정 변경 요청 승인

- **관련 액터**: ROLE-01(팀장), ROLE-02(팀원)
- **관련 PRD Must-have**: 일정 변경 요청 승인/거절(팀장 전용) / 승인·거절 처리 결과의 채팅 이력 자동 기록
- **관련 BR**: BR-05, BR-13

**사전조건**
- SC-06에서 생성된, 이서연의 "스프린트 계획 회의" 변경 요청(대기 상태)이 존재한다.

**기본 흐름**
1. 김팀장이 07-14 채팅 이력에서 이서연의 변경 요청 메시지를 확인하고, 대상 일정·현재 시간(11:00~12:00)·요청된 시간(14:00~15:00)을 검토한다.
2. 김팀장이 "승인" 버튼을 클릭한다.
3. 시스템은 요청 처리자가 해당 팀(테크팀)의 팀장 본인인지 서버 측에서 검증한다(BR-05). 검증을 통과하면 ScheduleChangeRequest 상태를 승인으로 전환하고, 동시에 대상 Schedule의 시작/종료 시각을 07-14 14:00~15:00으로 갱신한다. 승인 전까지 원래 일정(11:00~12:00)은 유효했다.
4. 시스템이 처리 결과를 07-14 채팅 이력에 자동으로 기록한다: "[시스템] 김팀장님이 이서연님의 일정 변경 요청을 승인했습니다. 스프린트 계획 회의가 07-14 14:00~15:00으로 변경되었습니다."(BR-13)

**대체/예외 흐름**
- **E1 (BR-05 위반, 팀원의 승인 시도 거부)**: 박준영이 승인 API를 직접 호출해 이서연의 요청을 승인하려 시도한다. 시스템은 박준영이 팀장이 아님을 확인하고 403 Forbidden "요청 승인 권한이 없습니다"를 반환하며 상태는 변경되지 않는다.
- **E2 (이미 처리된 요청의 중복 승인 방지)**: 김팀장이 이미 승인 처리를 완료한 동일 요청에 대해 승인 버튼을 다시 클릭한다(중복 클릭 등). 시스템은 ScheduleChangeRequest 상태가 이미 승인임을 확인하고 409 Conflict "이미 처리된 요청입니다"를 반환하며 추가 처리를 하지 않는다.

**사후조건**
- 이서연의 ScheduleChangeRequest 상태는 승인이다.
- "스프린트 계획 회의"는 07-14 14:00~15:00으로 갱신되어 있다.
- 07-14 채팅 이력에 처리 결과 시스템 메시지가 남아 있다.

**사후조건**

- "QA 결과 공유회" 일정은 원래 시간(07-18 14:00~15:00) 그대로 유지된다.

### SC-09. 동일 일정에 대한 복수 대기 요청 중 하나 승인 시 나머지 자동 거절

- **관련 액터**: ROLE-01(팀장), ROLE-02(팀원)
- **관련 PRD Must-have**: 동일 일정에 대한 복수 대기 요청 중 하나 승인 시 나머지 자동 거절
- **관련 BR**: BR-11, BR-13

**사전조건**
- "코드 프리즈 점검 회의"(2026-07-21(화) 09:00~10:00, 참여자: 이서연, 박준영, 최유진)가 존재한다.

**기본 흐름**
1. 이서연이 07-21 채팅에서 "코드 프리즈 점검 회의"에 대해 변경 요청 A를 제기한다: "오전 9시는 너무 이르니 11:00~12:00으로 옮겨주세요." 시스템은 이서연이 참여자임을 검증하고 요청 A를 대기 상태로 생성한다.
2. 10분 뒤, 박준영이 같은 일정에 대해 다른 내용의 변경 요청 B를 제기한다: "저는 오히려 하루 전날인 07-20 15:00로 당겼으면 합니다." 시스템은 박준영이 참여자임을 검증하고 요청 B도 대기 상태로 생성한다. 이 시점에 동일한 Schedule("코드 프리즈 점검 회의")을 대상으로 하는 대기 상태 ScheduleChangeRequest가 요청 A, 요청 B 두 건 동시에 존재한다(BR-11의 전제 상황).
3. 최유진은 별도 요청 없이 채팅을 지켜본다.
4. 김팀장이 두 요청을 모두 검토한 뒤, 먼저 제기된 요청 A(이서연)를 "승인"한다.
5. 시스템은 요청 A의 상태를 승인으로 전환하고 Schedule의 시각을 07-21 11:00~12:00으로 갱신한다. 이어서 시스템은 같은 Schedule을 대상으로 하는 다른 대기 상태 요청(요청 B)을 자동으로 조회하여 상태를 거절로 일괄 전환한다(BR-11).
6. 시스템이 두 건의 처리 결과를 각각 07-21 채팅 이력에 자동 기록한다(BR-13):
   - "[시스템] 이서연님의 요청이 승인되어 일정이 07-21 11:00~12:00으로 변경되었습니다."
   - "[시스템] 박준영님의 요청은 동일 일정에 대한 다른 요청이 먼저 승인되어 자동으로 거절되었습니다."
7. 박준영이 채팅 이력에서 자신의 요청이 자동 거절되었음을 확인한다.

**대체/예외 흐름**
- **A1 (대칭 케이스)**: 4단계에서 김팀장이 요청 A 대신 요청 B(박준영)를 먼저 승인했다면, 동일한 규칙에 따라 Schedule은 07-20 15:00로 갱신되고 요청 A(이서연)가 자동으로 거절 상태로 전환된다. 어느 요청이 먼저 승인되든 동일한 자동 거절 규칙이 대칭적으로 적용된다.
- **E1 (자동 거절된 요청의 재처리 방지)**: 김팀장이 이미 자동 거절된 요청 B를 뒤늦게 별도로 승인하려고 시도한다. 시스템은 요청 B의 상태가 이미 거절임을 확인하고 409 Conflict "이미 처리된 요청입니다"를 반환하며 추가 처리를 하지 않는다.

**사후조건**
- 요청 A는 승인, 요청 B는 거절 상태이며, "코드 프리즈 점검 회의"는 요청 A의 내용대로 07-21 11:00~12:00으로 변경되어 있다.
- 두 처리 결과 모두 07-21 채팅 이력에서 확인 가능하다.

### SC-10. 요청자 본인의 대기 중 변경 요청 취소

- SC-07에서 승인 완료된 이서연의 "스프린트 계획 회의" 변경 요청(상태=승인)이 존재한다.

**기본 흐름**
1. 최유진이 "고객 데모 준비"에 대해 변경 요청을 제기한다: "장비 점검 때문에 16:00~17:00으로 옮기고 싶습니다." 시스템은 참여자 검증(BR-10) 후 대기 상태 요청을 생성한다.
2. 잠시 후 장비 문제가 해결되어, 최유진은 기존 시간대로 진행이 가능하다고 판단한다.
3. 최유진이 07-16 채팅 이력에서 자신이 올린 요청 메시지 옆의 "요청 취소" 버튼을 클릭한다.
4. 시스템은 요청자 본인 여부와 현재 상태가 대기 중인지를 서버 측에서 검증한다(BR-12). 검증을 통과하면 ScheduleChangeRequest 상태를 취소로 전환한다.
5. 07-16 채팅 이력에 "[시스템] 최유진님이 변경 요청을 취소했습니다."가 기록되고, 해당 요청은 이후 김팀장의 처리 대상 목록에서 제외된다.

**대체/예외 흐름**
- **E1 (BR-12 위반, 이미 처리된 요청의 취소 시도 거부)**: 이서연이 이미 승인 처리가 완료된(SC-07) 자신의 "스프린트 계획 회의" 변경 요청에 대해 취소를 시도한다. 시스템은 해당 요청의 상태가 대기가 아니라 승인임을 확인하고 409 Conflict "이미 처리된 요청은 취소할 수 없습니다"를 반환하며 상태는 변경되지 않는다.
- **E2 (요청자 본인 여부 검증)**: 박준영이 최유진이 제기한 "고객 데모 준비" 변경 요청을 대신 취소하려고 시도한다. 시스템은 요청자가 박준영이 아님을 확인하고 403 Forbidden "본인이 제기한 요청만 취소할 수 있습니다"를 반환한다.

**사후조건**
- 최유진의 "고객 데모 준비" 변경 요청은 취소 상태이며, "고객 데모 준비" 일정은 원래 시간(15:00~16:00) 그대로 유지된다.
- 이서연의 이미 승인된 요청은 취소되지 않고 승인 상태를 유지한다.

### SC-11. 팀장이 1명뿐인 팀에서 제외/역할변경 시도 거부

- **관련 액터**: ROLE-01(팀장)
- **관련 PRD Must-have**: 팀 내 팀장 최소 1인 유지 제약(최소한의 검증 로직)
- **관련 BR**: BR-09

**사전조건**
- 테크팀에는 팀장 역할의 TeamMembership이 김철수 1건뿐이고, 이서연·박준영·최유진은 모두 팀원이다.

**기본 흐름**
1. 김팀장이 팀 설정 화면의 팀원 목록에서 자기 자신의 역할을 "팀원으로 변경"하도록 시도한다(예: 팀장 교체를 위한 사전 작업으로 실수 클릭).
2. 시스템은 역할 변경을 반영하기 전에, 대상 팀(테크팀) 내에서 역할=팀장인 TeamMembership의 수를 조회한다(BR-09).
3. 조회 결과 팀장이 김철수 1명뿐임을 확인한 시스템은 역할 변경 요청을 거부하고 오류를 반환한다: "팀에는 최소 1명의 팀장이 있어야 합니다. 다른 팀원을 먼저 팀장으로 지정한 후 다시 시도하세요." 역할은 변경되지 않는다.
4. 같은 상태에서 김철수가 본인을 팀에서 제외(탈퇴)하도록 시도해도, 시스템은 동일한 사유로 이를 거부한다.

**대체/예외 흐름**
- **A1 (팀장이 2명 이상일 때는 허용)**: 김팀장이 먼저 이서연을 팀장으로 승격시켜 테크팀에 팀장이 2명(김철수, 이서연)이 된 상태라면, 이후 김철수를 팀원으로 변경하거나 팀에서 제외하는 행위는 허용된다. 팀장이 0명이 되지 않기 때문이다.

**사후조건**
- 팀장이 1명뿐인 상태가 유지되는 한, 그 팀장의 역할 변경/제외는 항상 시스템에 의해 거부된다.
- 테크팀은 어떤 시점에도 팀장이 0명인 상태가 되지 않는다.

### SC-12. 채팅 메시지 작성 및 일자별 이력 조회

- **관련 액터**: ROLE-01(팀장), ROLE-02(팀원)
- **관련 PRD Must-have**: 팀 채팅 메시지 작성/조회, 일자 단위 이력 그룹핑 및 영구 보존
- **관련 BR**: BR-01, BR-06, BR-16

**사전조건**
- 테크팀이 존재하고 김철수·이서연·박준영·최유진이 소속되어 있다.
- SC-06, SC-07에서 07-14 채팅 이력에 변경 요청 및 처리 결과 메시지가 이미 쌓여 있다.

**기본 흐름**
1. 이서연이 테크팀 채팅의 2026-07-14 대화창에 일반 메시지를 작성한다: "오늘 스프린트 계획 회의 자료 미리 공유드립니다." 전송을 요청한다.
2. 시스템은 요청자가 인증된 사용자이며(BR-01) 테크팀 소속임을 확인한 뒤, Message(ENT-06)를 저장한다(작성자=이서연, 소속 팀=테크팀, 작성일시=2026-07-14 09:40, 유형=일반).
3. 07-14 채팅창에는 이 메시지가 앞서 발생한 변경 요청 메시지 및 처리 결과 시스템 메시지와 함께 작성일시 순으로 표시된다.
4. 다음날인 2026-07-15에 박준영이 채팅창에서 "이전 대화 보기"로 날짜를 07-14로 이동하면, 시스템은 작성일시를 기준으로 07-14에 속하는 모든 메시지(일반 대화, 변경요청, 처리 결과)를 일자 단위 이력으로 그룹핑해 반환한다.
5. 3주 뒤인 2026-08-01에 김팀장이 다시 07-14 이력을 조회해도, 메시지는 수정되거나 삭제되지 않고 영구 보존되어 그대로 확인된다(BR-06). Team CalTalk은 메시지 수정/삭제 기능을 제공하지 않는다.

**대체/예외 흐름**
- **E1 (BR-01/BR-16 위반, 팀 소속이 아닌 사용자의 조회 차단)**: 테크팀에 소속되지 않은 정다은(디자인팀 팀장)이 테크팀의 07-14 채팅 이력 조회를 시도한다. 시스템은 정다은에게 테크팀의 TeamMembership이 없음을 확인하고(BR-16) 403 Forbidden "해당 팀의 채팅에 접근할 수 없습니다"를 반환하며 메시지를 노출하지 않는다.

**사후조건**
- 07-14 채팅 이력에는 일반 메시지, 변경 요청 메시지, 처리 결과 메시지가 시간순으로 영구 보존되어 있으며, 팀 소속 사용자는 언제든 이를 조회할 수 있다.
