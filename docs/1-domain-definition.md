## 1. 문서 개요

### 1.1 목적
본 문서는 팀 단위 캘린더 기반 일정관리 및 채팅 애플리케이션 "Team CalTalk"의 핵심 도메인을 정의한다. 팀, 사용자 역할, 일정, 채팅, 일정 변경 요청 등 핵심 개념과 용어를 명확히 하여, 이후 진행될 요구사항 정의, 기능 명세, 데이터베이스 설계, API 설계 작업의 공통 기준(Ubiquitous Language)을 제공하는 것을 목적으로 한다.

### 1.2 범위
- 팀 생성 및 팀원 구성
- 팀 단위 일정(월/주/일)의 등록, 조회, 변경
- 일정과 결합된 팀 채팅 및 일자별 이력 관리
- 팀장/팀원 역할에 따른 일정 관련 권한 체계
- 채팅을 통한 일정 변경 요청 및 처리 프로세스

기술 스택, 화면 설계(UI/UX), 데이터베이스 스키마, API 명세 등 구현 수준의 결정은 본 문서의 범위에 포함하지 않으며, 개념 수준의 도메인 모델만을 다룬다.

## 2. 문제 정의 및 배경

| 문제 | 설명 |
|---|---|
| 일정 분산 | 팀 일정이 개인 캘린더, 메신저, 엑셀 등에 흩어져 있어 충돌 및 누락이 잦다. |
| 맥락 단절 | 일정과 그 일정에 관한 논의(대화)가 분리되어 있어, 일정이 왜 그렇게 정해졌는지 맥락을 되짚기 어렵다. |
| 통제 부재 | 팀장이 팀원 개인의 일정에 개입하거나 조율할 마땅한 수단이 없다. |

Team CalTalk은 캘린더와 채팅을 하나의 화면에서 함께 제공하여, 팀장이 팀 일정을 일원화된 권한으로 관리하고, 팀원은 일정 관련 논의와 변경 요청을 캘린더와 같은 맥락에서 처리할 수 있도록 한다.

## 3. 핵심 용어 정의 (Ubiquitous Language)

| ID | 용어 | 정의 |
|---|---|---|
| TERM-01 | 사용자(User) | 인증을 거쳐 시스템을 이용하는 주체. 하나 이상의 팀에 소속될 수 있다. |
| TERM-02 | 팀(Team) | 함께 일정을 공유하는 사용자들의 집합. 모든 일정과 채팅은 특정 팀에 귀속된다. |
| TERM-03 | 팀장(Team Leader) | 팀 내에서 팀 일정에 대한 생성/수정/삭제 권한과 일정 변경 요청에 대한 승인/거절 권한을 가진 역할. |
| TERM-04 | 팀원(Team Member) | 팀에 소속되어 팀 일정을 조회하고, 채팅에 참여하며, 자신이 참여자로 지정된 일정에 대해 변경을 요청할 수 있는 역할. |
| TERM-05 | 팀 일정(Schedule) | 팀에 귀속되어 팀원들에게 공유되는 일정. 제목, 일시, 관련 팀원 등의 정보를 가진다. 본 서비스에서 다루는 일정은 원칙적으로 팀 일정이며, 팀에 속하지 않는 개인 전용 캘린더는 두지 않는다(8장 참조). |
| TERM-06 | 일정 참여자(Schedule Participant) | 특정 팀 일정과 관련이 있는 팀원. "자신과 관련된 일정"의 판단 기준이 되며, 팀원이 일정 변경 요청(TERM-09)을 제기할 수 있는 대상 범위를 결정한다(BR-10). |
| TERM-07 | 채팅(Chat) | 팀 단위로 이루어지는 대화 공간으로, 일정에 관한 논의와 일정 변경 요청이 이루어지는 곳. 팀별로 일자(날짜) 단위 이력을 가진다. |
| TERM-08 | 메시지(Message) | 채팅에서 작성되는 개별 대화 단위. 일반 대화 메시지, 일정 변경 요청 메시지, 시스템이 처리 결과를 알리기 위해 자동 생성하는 시스템 메시지로 구분된다(ENT-06). |
| TERM-09 | 일정 변경 요청(Schedule Change Request) | 팀원이 자신이 일정 참여자(TERM-06)로 지정된 팀 일정에 대해 채팅을 통해 제기하는 변경 제안. 제목, 시작/종료 일시 등 제안하는 일정 값을 구조화된 필드로 포함할 수 있다(ENT-07). 상태는 대기/승인/거절/취소 중 하나를 가지며, 팀장의 승인이 있어야 실제 일정에 반영된다(BR-05, BR-10~BR-13). |
| TERM-10 | 인증(Authentication) | 사용자가 시스템 이용을 위해 로그인 등을 통해 본인임을 증명하는 절차. 모든 기능 이용의 전제 조건이다. |

## 4. 액터(사용자 역할) 정의와 권한

### 4.1 액터

| ID | 액터 | 설명 |
|---|---|---|
| ROLE-01 | 팀장 | 팀 내 일정 관리의 최종 권한을 가진 사용자. 팀원의 일정 변경 요청을 검토하고 승인/거절한다. |
| ROLE-02 | 팀원 | 팀 일정을 조회하고 채팅에 참여하며, 일정 변경이 필요한 경우 요청을 제기하는 사용자. |

동일한 사용자라도 소속된 팀이 다르면 팀별로 역할(팀장/팀원)이 다를 수 있다. 즉 역할은 사용자 고유 속성이 아니라 "팀-사용자 소속 관계"에 종속된 속성이다.

### 4.2 역할별 권한 매트릭스

| 대상 / 행위 | 팀장 | 팀원 | 근거 규칙 |
|---|---|---|---|
| 팀 생성 | O | - | BR-15 |
| 팀원 초대/추가 | O | - | BR-14 |
| 팀원 제외 | O | - | BR-09 |
| 팀 일정 조회(월/주/일) | O | O | BR-03, BR-16 |
| 팀 일정 생성 | O | X | BR-02 |
| 팀 일정 수정 | O | X | BR-02, BR-03 |
| 팀 일정 삭제 | O | X | BR-02, BR-03 |
| 채팅 메시지 작성/조회 | O | O | BR-06, BR-16 |
| 일정 변경 요청 제기 | X (직접 수정 가능하므로 불필요) | O (자신이 참여자인 일정에 한함) | BR-04, BR-10 |
| 일정 변경 요청 취소 | - | O (자신이 제기한 대기 중 요청에 한함) | BR-12 |
| 일정 변경 요청 승인/거절 | O | X | BR-05, BR-11 |

핵심 원칙: 팀 일정에 대한 쓰기 권한(생성/수정/삭제)은 팀장에게 배타적으로 귀속되며, 팀원은 읽기 권한만 가진다. 팀원이 일정 변경을 원할 경우 반드시 채팅을 통한 "요청"이라는 간접적 경로만 허용된다.

## 5. 핵심 도메인 개념/엔티티와 관계

### 5.1 개념 모델 (요약)

```
User ──(소속, 역할: 팀장/팀원)── Team
Team ──(1:N)── Schedule
Schedule ──(N:M, 선택)── User   [일정 참여자]
Team ──(1:N)── Message   [일자 단위로 그룹핑되어 조회]
Message ──(0..1:1)── ScheduleChangeRequest
ScheduleChangeRequest ──(N:1)── Schedule  [변경 대상]
ScheduleChangeRequest ──(N:1)── User      [요청자, 팀원]
ScheduleChangeRequest ──(N:1)── User      [처리자, 팀장 / 승인 시]
ScheduleChangeRequest ──(처리 시 시스템이 자동 생성)── Message   [시스템 메시지]
```

### 5.2 엔티티 정의

| ID | 엔티티 | 설명 | 주요 속성(개념 수준) |
|---|---|---|---|
| ENT-01 | User | 시스템 이용자 | 이름, 이메일 등 식별정보, 인증정보 |
| ENT-02 | Team | 일정과 채팅을 공유하는 조직 단위 | 팀 이름 |
| ENT-03 | TeamMembership | User와 Team의 소속 관계 | 역할(팀장/팀원), 가입일 |
| ENT-04 | Schedule | 팀에 귀속된 일정 | 제목, 설명, 시작/종료 일시, 작성자(팀장), 소속 팀 |
| ENT-05 | ScheduleParticipant | Schedule과 관련된 팀원 | Schedule 참조, User 참조 |
| ENT-06 | Message | 채팅 메시지 | 작성자(User, 시스템이 자동 생성하는 메시지의 경우 nullable), 소속 팀, 작성일시(→ 일자 단위 이력의 기준), 본문, 메시지 유형(일반/변경요청/시스템 처리결과) |
| ENT-07 | ScheduleChangeRequest | 팀원이 제기하는 일정 변경 제안 | 대상 Schedule, 요청자(팀원), 제안 제목(선택), 제안 시작 일시(선택), 제안 종료 일시(선택), 요청 사유(자유 텍스트), 상태(대기/승인/거절/취소), 처리자(팀장), 처리일시, 처리 결과(채팅 이력 기록 여부) |

### 5.3 관계 설명
- 하나의 User는 여러 Team에 소속될 수 있으며, 팀마다 역할(팀장/팀원)이 다를 수 있다.
- 하나의 Schedule은 반드시 하나의 Team에 속하며, 여러 Team이 하나의 일정을 공유하지 않는다.
- Schedule은 0명 이상의 ScheduleParticipant(팀원)를 가질 수 있으며, 이는 "자신과 관련된 일정"을 판별하는 기준이 된다.
- Message는 반드시 하나의 Team과 작성일시를 가지며, 채팅 이력은 이 작성일시를 기준으로 일자 단위로 묶여 조회된다.
- ScheduleChangeRequest는 특정 Message(요청을 표현하는 메시지)와 연결되며, 하나의 Schedule을 변경 대상으로 지정한다.
- 하나의 Schedule에는 동시에 여러 개의 대기(pending) 상태 ScheduleChangeRequest가 존재할 수 있다. 팀장이 그중 하나를 승인하면 같은 Schedule을 대상으로 하는 나머지 대기 중 요청은 자동으로 거절 상태로 전환된다(BR-11).
- ScheduleChangeRequest는 요청자 본인에 의해 대기 상태에서 취소될 수 있으며, 취소된 요청은 팀장의 처리 대상에서 제외된다(BR-12).
- ScheduleChangeRequest가 승인 또는 거절되면, 시스템은 그 처리 결과를 알리는 시스템 메시지(Message, 유형=시스템 처리결과)를 자동 생성하여 해당 Team의 채팅 이력에 추가한다(BR-13). 이 메시지는 사람이 아닌 시스템이 작성하므로 작성자가 없다(nullable).

## 6. 핵심 유스케이스/시나리오

### 6.1 공통 (팀장/팀원 모두)
- 로그인하여 인증 후 서비스 이용
- 팀 캘린더를 월/주/일 단위로 조회
- 캘린더와 채팅창을 함께 열람하며 특정 일자의 일정과 관련 대화 확인
- 팀 채팅에 메시지 작성 및 일자별 이력 조회

### 6.2 팀장
1. 팀을 생성하고 팀원을 초대한다.
2. 팀 일정을 등록한다(제목, 일시, 관련 팀원 지정).
3. 등록된 팀 일정을 수정하거나 삭제한다.
4. 채팅에서 팀원이 올린 일정 변경 요청을 확인한다.
5. 일정 변경 요청을 승인하여 해당 일정에 반영하거나, 거절한다.

### 6.3 팀원
1. 팀 캘린더에서 팀 일정을 조회한다(수정 불가).
2. 캘린더를 보다가 자신과 관련된 일정에 변경이 필요함을 인지한다.
3. 해당 일자의 채팅에서 일정 변경을 요청하는 메시지를 작성한다.
4. 팀장의 승인/거절 결과를 채팅 이력을 통해 확인한다.
5. 승인된 경우, 갱신된 일정을 캘린더에서 확인한다.

## 7. 비즈니스 규칙 및 제약사항

| ID | 규칙 |
|---|---|
| BR-01 | 모든 기능은 인증된 사용자만 이용할 수 있다. 미인증 사용자는 어떠한 팀/일정/채팅 정보에도 접근할 수 없다. |
| BR-02 | 팀 일정의 생성, 수정, 삭제 권한은 해당 팀의 팀장에게만 있다. |
| BR-03 | 팀원은 팀 일정에 대해 조회 권한만 가지며, 어떠한 경우에도 팀 일정을 직접 수정할 수 없다. |
| BR-04 | 팀원이 일정 변경을 원하는 경우, 반드시 채팅을 통한 일정 변경 요청(TERM-09) 절차를 거쳐야 한다. |
| BR-05 | 일정 변경 요청은 팀장의 승인이 있어야 실제 Schedule에 반영된다. 요청에 제안 값(ENT-07의 제안 제목/제안 시작·종료 일시)이 포함된 경우 승인 시 그 값이 Schedule에 그대로 반영되며, 승인 전까지 원래 일정은 유효하다. |
| BR-06 | 채팅 메시지는 팀 단위, 작성일자 단위로 이력이 관리되며 영구 보존된다. 메시지 수정/삭제 기능은 제공하지 않는다(8장 Out of Scope). 사용자는 언제든 과거 일자의 채팅 이력을 조회하여 일정 관련 논의 맥락을 확인할 수 있어야 한다. |
| BR-07 | 하나의 팀 일정은 하나의 팀에만 속하며, 팀 간에 공유되지 않는다. |
| BR-08 | 사용자는 여러 팀에 동시에 소속될 수 있고, 팀별로 역할이 다를 수 있다(한 팀의 팀장이 다른 팀에서는 팀원일 수 있음). |
| BR-09 | 하나의 팀에는 최소 1명 이상의 팀장이 존재해야 한다. 팀에 팀장이 1명뿐인 경우, 해당 팀장의 역할을 팀원으로 변경하거나 팀에서 제외하는 행위는 허용되지 않는다. |
| BR-10 | 팀원은 자신이 일정 참여자(ENT-05, ScheduleParticipant)로 지정된 일정에 대해서만 일정 변경 요청을 제기할 수 있다. 자신이 참여자가 아닌 일정에 대한 변경 요청은 허용되지 않는다. |
| BR-11 | 하나의 Schedule에는 동시에 여러 개의 대기(pending) 상태 ScheduleChangeRequest가 존재할 수 있다. 팀장이 그중 하나를 승인하면, 같은 Schedule을 대상으로 하는 나머지 대기 중 요청은 시스템에 의해 자동으로 거절 상태로 전환된다. |
| BR-12 | 요청자는 자신이 제기한 ScheduleChangeRequest가 대기 상태인 동안에만 이를 취소할 수 있다. 이미 승인되었거나 거절된 요청은 취소할 수 없다. |
| BR-13 | 일정 변경 요청이 승인 또는 거절되면, 그 처리 결과는 해당 요청이 제기된 채팅 이력에 기록되어 요청자가 확인할 수 있어야 한다. 이는 8장에서 범위 외로 정의한 "알림 채널의 세부 구현"과는 별개로, 본 서비스가 최소한으로 보장해야 하는 결과 확인 경로다. |
| BR-14 | 팀장은 이미 가입된 사용자를 이메일로 검색하여 즉시 팀원으로 추가한다. 초대 발송이나 수락 대기와 같은 별도 절차는 존재하지 않으며, 추가 즉시 TeamMembership이 생성된다. 대상 사용자가 아직 가입하지 않은 경우, 해당 사용자가 먼저 회원가입을 완료한 뒤에만 추가할 수 있다. |
| BR-15 | 팀을 생성한 사용자는 해당 팀의 팀장으로 자동 등록된다. |
| BR-16 | 사용자는 자신이 소속되지 않은 팀의 일정, 채팅, 팀 정보에 접근할 수 없다. |

## 8. 도메인 경계 및 Out of Scope

본 도메인 정의서 및 이를 기반으로 한 1차 범위에서는 다음 항목을 다루지 않는다.

| 제외 항목 | 비고 |
|---|---|
| 결제/구독 관리 | 유료화 모델은 별도 검토 대상 |
| 알림 채널의 세부 구현(이메일, 푸시, SMS 등) | "알림이 필요하다"는 사실만 인지하며, 발송 수단/구현 방식은 범위 외 |
| 외부 캘린더 연동(Google Calendar, Outlook 등) | 향후 확장 검토 대상 |
| 팀에 속하지 않는 개인 전용 캘린더 | 본 서비스의 모든 일정은 팀 일정을 전제로 하며, 개인 일정 관리 기능은 범위 외 |
| 채팅 부가 기능(파일 첨부, 이모지 반응, 메시지 수정/삭제 등) | 텍스트 기반 대화와 일정 변경 요청 흐름만 다룸 |
| 조직/부서 등 팀 상위 구조, 여러 팀 간 계층 | 팀은 최상위 독립 단위로 취급 |
| 시스템 관리자(운영자) 역할 및 관리자 콘솔 | 팀장/팀원 2개 역할만 정의 |
| 다국어 지원, 타임존 세부 처리 | 구현 단계에서 별도 검토 |
| 특정 클라이언트 플랫폼(웹/모바일 등) 및 UI/UX 세부사항 | 기술 중립적 도메인 정의로 한정 |
| 데이터베이스 스키마, API 명세 등 구현 상세 | 후속 산출물에서 다룸 |
