## 3. 발견 및 수정된 버그

- **일정 상세 조회 엔드포인트 라우팅 누락** (SC-04 기본흐름 4단계 관련)
  - 증상: `swagger/swagger.json`에는 `GET /api/teams/{teamId}/schedules/{scheduleId}`(참여자 포함 상세 조회, SC-04 근거)가 문서화되어 있었으나, `src/modules/team-schedule/schedule.routes.js`에는 해당 라우트가 전혀 등록되어 있지 않았다. 실제 호출 시 `notFoundHandler`가 걸려 `404 {"error":{"code":"NOT_FOUND","message":"요청한 경로를 찾을 수 없습니다"}}`를 반환했다(문서화된 스펙과 실제 동작 불일치).
  - 원인: `schedule.service.js`에 이미 존재하던 `getScheduleById()`는 change-request 모듈(BE-18)의 내부용으로만 쓰였고, 참여자 목록까지 포함한 상세 조회를 노출하는 라우트/서비스 함수가 별도로 구현되지 않았다.
  - 수정: `backend/src/modules/team-schedule/schedule.service.js`에 `getScheduleDetail(teamId, scheduleId)`를 추가(대상 팀 소속 여부 검증 후 404 처리, 참여자 목록 조회는 기존 `getUsersByIds`/`getParticipantUserIds` 재사용)하고, `backend/src/modules/team-schedule/schedule.routes.js`에 `GET /:teamId/schedules/:scheduleId` 라우트를 추가해 `teamAccessMiddleware`(BR-16)를 통과한 뒤 이 함수를 호출하도록 배선했다.
  - 재검증: 수정 후 동일 요청이 `200`과 함께 `{ ..., participants: [...] }`를 정상 반환함을 확인(QA 스크립트 재실행 결과 56/56 PASS로 전환).

이 외 BR-xx/SC-xx 관련 실제 동작 불일치는 발견되지 않았다. (에러 메시지 문구가 시나리오 문서의 예시 문장과 완전히 동일하지 않은 경우가 일부 있었으나 — 예: 로그인 실패 메시지 "이메일 또는 비밀번호가 일치하지 않습니다" vs 문서 예시 "...올바르지 않습니다" — 의미가 동일한 표현 차이이며 상태 코드는 모두 문서와 일치해 버그로 판단하지 않았다.)
